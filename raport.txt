RAPPORT D'ANALYSE DU PROJET : GESTION DE TRANSPORT UNIVERSITAIRE

1. INTRODUCTION
Ce projet est une application de gestion de transport universitaire développée en Java. Elle permet de gérer les utilisateurs (étudiants, chauffeurs, administrateurs), les bus, les trajets et les incidents. L'application utilise une architecture en couches séparant les données, la logique métier et l'interface utilisateur.

2. STRUCTURE DU PROJET ET GESTION DES FICHIERS
Le projet est organisé selon une structure modulaire claire :

- src/
  - Main.java : Point d'entrée de l'application.
  - Model/ : Contient les classes représentant les entités du domaine.
  - DATA/ : Contient les classes d'accès aux données (DAO) pour interagir avec la base de données MySQL.
  - Service/ : Contient la logique métier faisant le lien entre l'interface et les données.
  - UI/ : Contient les interfaces graphiques (Swing) pour les différents acteurs.

3. ANALYSE DÉTAILLÉE DES PACKAGES

3.1 Package Model (Les Entités)
Ce package définit les objets manipulés par l'application. Le modèle ne repose pas sur l'héritage pour les utilisateurs, mais sur la composition.

- CompteUtilisateur : Classe centrale pour l'authentification. Elle contient les informations de connexion (id, email, motDePasse, role).
- Admin : Représente un administrateur. Il est lié à un CompteUtilisateur et gère les listes des autres entités (Bus, Chauffeurs, etc.).
- Etudiant : Représente un étudiant. Il est lié à un CompteUtilisateur et possède des attributs spécifiques : numCarte, arretPrincipal, statutPaiement.
- Chauffeur : Représente un chauffeur. Il est lié à un CompteUtilisateur et possède un attribut spécifique : typePermis.
- Bus : Représente un véhicule (id, matricule, marque, capacite).
- Trajet : Représente une course (pointDepart, pointArrivee, heureDepart) associée à un Bus et un Chauffeur.
- Incident : Représente un problème signalé (description, date, status) lié à un Bus et un Chauffeur.
- IncidentDetails : Classe de vue qui agrège les informations d'un incident avec les détails du bus et du chauffeur.

3.2 Package DATA (Accès aux Données)
Ce package gère la persistance via JDBC.
- ConnexionBDD : Gère la connexion unique à la base de données 'transport_db'.
- CompteUtilisateurDAO : Gère l'authentification et le CRUD pour les comptes utilisateurs.
- EtudiantDAO, ChauffeurDAO, BusDAO, TrajetDAO, IncidentDAO : Gèrent le CRUD pour leurs entités respectives. Le TrajetDAO gère notamment les jointures pour reconstituer les objets Bus et Chauffeur complets.

3.3 Package Service (Logique Métier)
Ce package agit comme une façade pour l'UI, orchestrant les appels aux DAO.
- CompteUtilisateurService : Orchestre l'authentification et la gestion des comptes.
- BusService, ChauffeurService, EtudiantService, TrajetService, IncidentService : Servent de relais vers les DAO correspondants pour appliquer la logique métier.

3.4 Package UI (Interface Utilisateur)
- LoginFrame : Fenêtre de connexion.
- AdminDashboard : Tableau de bord pour l'administrateur.
- StudentDashboard : Interface pour les étudiants.
- ChauffeurDashboard : Interface pour les chauffeurs.

4. DIAGRAMME DE CLASSES ET RELATIONS

Le diagramme de classes détaillé, incluant les attributs, méthodes et relations, a été généré au format PlantUML et est disponible dans le fichier suivant : `src/explication .txt`.

Relations clés :
- Composition/Association : Admin, Etudiant, et Chauffeur sont chacun associés à un CompteUtilisateur. Ce n'est pas un héritage.
- Association : Un Trajet est associé à un Bus et à un Chauffeur.
- Association : Un Incident est lié à un Bus et à un Chauffeur via leurs identifiants.

5. CONCLUSION
L'application présente une architecture solide et extensible. La séparation claire entre le Modèle, les Données (DAO), les Services et l'UI facilite la maintenance et l'évolution du projet. L'utilisation de DAO permet d'abstraire efficacement la logique d'accès à la base de données.
